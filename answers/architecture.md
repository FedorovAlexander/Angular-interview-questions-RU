## <a name="ngrx-store"></a>Приведите хороший пример когда нужно использовать ngrx/store?

- Когда в приложении мы имеем 5 или 10 уровней дерева компонентов. Мы передаем переменные через `@Input` и конечный компоненты знают, что нужно делать переданными данными. Но для промежуточных компонентов эти данные кажутся лишними. Компоненты становится сложнее переиспользовать. В случае нескольких уровней вложенности компонентов реагировать на события также становится проблемой. Все это усложняет масштабирование приложения.
  <br/>
- Когда данные могут изменятся как на клиенте так и на сервере. Store в таком случае будет являться единственным источником истины.

<br/>

## <a name="race-condition"></a>Что такое "race condition" и какие баги могут быть связаны с этим? Как с ними справиться?

Состояние гонки возникает, когда два или более потоков могут получить доступ к общим данным, и они пытаются изменить их одновременно.

Пример. У нас есть список фильмов. По клику на название показывается описание фильма. Мы кликаем на "Форест Гамп", а потом сразу на "Интерстеллар". В идеальном мире ответы от сервера приходят по очереди. Но если описание к "Интерстеллар" придет первым, то мы получим баг.

Решить эту проблему можно с помощью `Subject` и `switchMap`. Вместо того, чтобы при каждом клике создавать новый независимый Observable мы создаем один Observable, который синхронизирует запросы. Оператор `switchMap` подписывается только на последний запрос. Если происходит новый запрос, он подписывается на него и отписывается от предыдущего.

<br/>

## <a name="smart-dumb"></a>Разница между умным и презентационным компонентом? Приведите пример использования? Назовите преимущества?

Умный компонент запрашивает данные у сервиса и знает, что это за данные и какой у них тип. Обрабатывает события, которые были посланы из презентационного компонента.

Презентационный компонент отображает данные, которые он получает через `@Input` из умного компонента, модифицирует их и возвращает умному компоненту через `@Output`. Не в курсе, откуда пришли эти данные. Любые события презентационного компонента эмитятся в умный компонент. При этом презентационный компонент не знает, кто получит эти события и кто будет их обрабатывать.

**Пример**: <br/>
Список пользователей - умный компонент, карточка пользователя - презентационный.

**Преимущества**:

- Презентационные компоненты легко переиспользовать
- Разделение ответственности (UI отдельно от логики)
- Читаемость кода лучше

<br/>

## <a name="shared"></a>Что такое Shared модуль?

Shared модуль позволяет организовать код. В нем находятся директивы, пайпы и компоненты которые используются в других модулях приложения. Мы импортируем один Shared модуль вместо нескольких импортов директив, пайпов и компонентов.

## <a name="signals-vs-ngrx"></a>Когда стоит строить архитектуру состояния приложения на Angular Signals, а когда лучше использовать NgRx/RxJS?

Angular Signals хорошо подходят для локального и средне-сложного состояния, когда:

- Большая часть состояния относится к UI (фильтры, выбранные элементы, форма, текущие вкладки и т.п.).
- Нет жёстких требований к time-travel дебаггингу, сложным сайд-эффектам и строгому однонаправленному потоку данных.
- Хочется уменьшить количество boilerplate и избавиться от большого количества actions/reducers/effects.

NgRx/RxJS по‑прежнему целесообразны, когда:

- Приложение большое, с множеством независимых доменов и сложными бизнес‑правилами.
- Нужны явные и тестируемые сайд‑эффекты (effects), time-travel debugging, строгий audit изменений состояния.
- Есть несколько источников изменений (WebSocket, несколько HTTP‑потоков, worker‑ы и т.д.), которые проще описывать операторами RxJS.

На практике часто используют гибридный подход: локальное состояние и часть feature‑состояния строятся на Signals, а критичный доменный стор — на NgRx.

<br/>

## <a name="signals-architecture"></a>Как спроектировать слои приложения при использовании Signals для состояния?

Обычно выделяют несколько уровней:

- **Компоненты** — используют сигналы для локального UI‑состояния (открыт ли сайдбар, выбранный таб, текущий фильтр).
- **Signal‑сервисы** — сервисы, внутри которых хранятся сигналы и предоставляются методы для изменения состояния. Они инжектятся в компоненты и другие сервисы.
- **Глобальный стор на сигналах** — при необходимости можно создать «мини‑стор», где состояние хранится в сигналах, а операции изменения — в методах сервиса (без actions/reducers).

Важно:

- Сигналы должны описывать состояние, а не действия (никаких `isLoadingAndUserClickedThreeTimes`).
- Писать чистые методы изменения состояния (по сути, маленькие reducers), чтобы их было легко тестировать.

<br/>

## <a name="signals-migration"></a>Как мигрировать существующее приложение с `@Input/@Output` и RxJS к Signals, не переписывая всё сразу?

Поэтапная стратегия:

1. **Компоненты**: начать с новых и наиболее часто изменяемых компонентов — использовать `standalone: true`, `input()` / `output()` сигналы вместо `@Input/@Output`.
2. **Локальное состояние**: заменить внутри компонентов `BehaviorSubject` / `Subject` на сигналы там, где это чисто UI‑состояние.
3. **Сервисы**: постепенно выносить часть состояния в signal‑сервисы (с сигнальными свойствами вместо Subjects), оставляя RxJS для внешних источников (HTTP, WebSocket).
4. **Интеграция с NgRx**: по необходимости использовать сигналы поверх селекторов `store.select(...)`, превращая поток в сигнал (например, через utility‑функции или специальные адаптеры).

Ключевая идея — не трогать всё приложение сразу, а двигаться итеративно, сохраняя существующий RxJS/NgRx там, где он уже хорошо работает.

<br/>

## <a name="signals-antipatterns"></a>Какие анти‑паттерны при использовании Signals вы бы выделили?

- Хранение всего состояния приложения в одном огромном сигнале (сложно обновлять, плохо контролировать зависимости, риск ненужных перерисовок).
- Дублирование одного и того же состояния одновременно в RxJS‑стримах и в сигналах без явной необходимости.
- Помещение сложной асинхронной логики внутрь `computed()` вместо использования сервисов и/или RxJS там, где это уместно.
- Массовое использование императивных вызовов `set`/`update` из разных мест, вместо централизованных методов изменения состояния в сервисах.

Хорошая практика — держать сигналы ближе к данным и UI, а сложную оркестрацию асинхронщины продолжать решать через RxJS/NgRx.
